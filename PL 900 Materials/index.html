Security, Governance & Administration

Power Platform is built on a layered security model. Understanding environments, admin controls, and governance policies is essential for both the PL-900 exam and real-world deployment.

Microsoft Power Platform Security Model

The Power Platform security model ensures protection of data and resources from unauthorised access or modification. It is structured across three distinct layers, each serving a critical role in governance and access control.

1

Tenant-Level Security

Managed via Microsoft 365 Admin Center. Controls user accounts, service access, and tenant-wide settings using Azure Active Directory.

2

Environment-Level Security

Managed via Power Platform Admin Center. Controls access to environments, apps, and data with isolation, permissions, and governance strategies.

3

Data-Level Security

Enforced within Dataverse. Includes resource permissions, connector access, data loss prevention policies, and on-premises data gateway support.

Environments in Power Platform

An environment in Power Platform is a designated space to manage and store an organisation's business data, apps, chatbots, and flows. It acts as a container to separate solutions based on roles, security needs, or target audiences.

⚙️

Usage Scenarios

  • Build apps or chatbots in a single environment
  • Separate test and production versions into distinct environments
  • Create environments for specific teams or departments with tailored data and apps
  • Establish environments for global branches to localise access and governance
🛡️

Governance Benefits

  • Enhances data isolation and access control
  • Supports lifecycle management across development stages
  • Aligns with organisational structure and compliance needs

Types of Environments in Power Platform

Power Platform supports five main environment types, each serving a distinct purpose in the app lifecycle and organisational structure.

1

Production

Default environment for deploying finished apps and solutions. Intended for end-user access and live operations.

2

Sandbox

Non-production environment for development and testing. Isolated from production data and impact.

3

Trial

Temporary environment valid for 30 days. Ideal for exploring Power Platform capabilities before licensing.

4

Developer

Non-production environment for individual developers with full access to Power Platform features for building and testing solutions.

5

Default

Automatically created for each tenant upon provisioning. Cannot be deleted, but can be renamed and customised.

Feature Traditional Databases Microsoft Dataverse
Architecture Relational tables with defined relationships No-code/Low-code platform using EAV data model

Types of Tables in Dataverse

Dataverse supports three main types of tables, each with distinct customisation and management characteristics.

1

Standard Tables

Pre-installed out-of-the-box tables that come with every Dataverse instance. (e.g. Account, Contact, Task, User)

  • Can be customised if marked as customisable
  • Includes customisable tables from managed solutions
  • Editable by users with appropriate permissions

Standard tables form the foundation of every Dataverse environment and cover the most common business data scenarios out of the box.

2

Managed Tables

Imported through managed solutions into a Dataverse environment.

  • Cannot be customised - locked to preserve solution integrity
  • Controlled entirely by the solution publisher
  • Protects intellectual property and solution consistency

Purpose: Used when organisations want to distribute applications while preventing unauthorised modifications. Managed tables ensure that distributed solutions remain stable and behave as intended by the publisher.

3

Custom Tables

Created directly within Dataverse or imported from unmanaged solutions. Fully customisable to meet any organisational need. (e.g. Employee Requests, Leave Management, Asset Tracking, Travel Requests)

  • Fully customisable - modify structure freely
  • Users with appropriate privileges can alter the schema
  • Ideal for organisation-specific business requirements

Relationships in Dataverse

In Microsoft Dataverse, relationships define how tables are connected using shared data. They enable structured, consistent links between records across tables.

🔑

Key Concepts

  • A foreign key in the primary table references the primary key of a related table
  • Relationships ensure consistent updates across linked records
  • Critical for building complex data models that support real-world business scenarios
✨

Benefits

  • Improves efficiency, accuracy, and insight
  • Enables meaningful data organisation and navigation
  • Supports scalable relational logic across apps and workflows

Relationship Types

One-to-Many (1:N)

One record in the primary table links to many records in the related table.

Customer (1)
➔
Orders (N)
Shipments
Invoices
Many-to-One (N:1)

Many records in the primary table link to one record in the related table. Viewed from the child table looking up to its parent.

Order A
Order B
Order C
➔
Customer (1)
Many-to-Many (N:N)

Many records in both tables link to many records in the other table.

Orders (N)
↔
Products (N)

Example: A customer can place multiple orders, and each order can include multiple products.

ℹ️ Dataverse supports these three relationship types (1:N, N:1, N:N) to model real-world business scenarios.

Dataflows - Incremental Refresh

What Is Incremental Refresh? Only changed data is loaded during each refresh cycle, rather than reloading the entire dataset.

⚡

Benefits

  • Faster refresh performance
  • Reduced processing overhead
  • Better scalability for large datasets
📅

Scheduling Options

  • Daily: Refresh every day automatically
  • Weekly: Refresh on a set day each week
  • On-Demand: Trigger manually when needed
ℹ️ Transformed data is loaded into Dataverse tables and can be used in Apps, Reports, and Dashboards.
